Phase 0: The Onboarding Interview

Before a single line of code is written, the Student conducts a structured interview with you. Goal: a complete project brief that aligns all agents toward the same objective. The brief is saved as project_brief.md and is the shared source of truth for all phases.

1
Phase

🎓 The Student

Prompt Architect — Executing Agent

The Student receives the project mandate and builds the first version of the application. They first conduct the onboarding interview, generate the project_brief.md from it, and then implement the code. They follow the 3-layer architecture: directives define the goal, they orchestrate, execution scripts handle the deterministic work.

Onboarding Interview Code Generation project_brief.md 3-Layer Architecture

Inputs & Outputs

→ Input: User vision (free text)
→ Output: project_brief.md
→ Output: Working App v1
→ Output: Updated Directives

Core Principle

⚠ Deterministic scripts instead of probabilistic code
2
Phase

🔎 The Examiner

Project Supervisor — Agentic Agent 1

The Examiner is the strict reviewer. They have the original project_brief.md in their instructions and know exactly what was required. They open the app in Chrome, systematically click through every function, take screenshots, mark errors with arrows, and write precise correction prompts for the Student. Fresh context window — no context pollution.

Chrome Automation Screenshot Analysis Error Annotation Correction Prompts

Inputs & Outputs

→ Input: project_brief.md
→ Input: App URL / local server
→ Output: Annotated screenshots
→ Output: Correction prompts
→ Output: examiner_report.md

Core Principle

⚠ Fresh context — no bias from Student's conversation
🔄 REVIEW LOOP — Repeat Phase 2 ↔ Phase 3 until error-free
3
Phase

🛠 The Revision

Student Round 2+ — Iterative Fixer

The same Student receives the Examiner's correction prompts and works through them systematically. Every fix is documented. After each round the Examiner reviews again. The loop only ends when the Examiner finds zero errors. Every correction flows back into the directives (Self-Annealing).

Fix-It Prompts Iterative Self-Annealing Documentation

Inputs & Outputs

→ Input: examiner_report.md
→ Input: Correction prompts
→ Output: Fixed app
→ Output: revision_log.md

Exit Criterion

✅ Examiner confirms: 0 errors, all features per brief implemented
4
Phase

🎨 The Creative Director

Subject Expert — Agentic Agent 2

The Creative Director is the experienced professor. They know the project_brief.md, use the app like a real user in Chrome, and evaluate holistically: What's missing? Where is the UX unnecessarily complex? Which features would multiply the value by ten? They present their suggestions to the User (not the Student). The user decides which improvements get implemented — and the cycle starts again.

Holistic Evaluation UX Analysis 10x Improvements Human-in-the-Loop

Inputs & Outputs

→ Input: project_brief.md
→ Input: Finished, bug-free app
→ Output: Improvement suggestions
→ Output: New prompts (user-approved)

Core Principle

⚠ Only 10x improvements — no marginal optimizations
🔄 CREATIVE LOOP — User selects improvements → Back to Phase 1
